# Spec 评审清单

按下面三方面问题对 spec **逐条**出具评审意见。每条标 ✓ / ✗ / ⚠，✗ 项必须解决才能冻结。

---

## 一、完整性

> 检查 spec 是否"说全了"。

- [ ] 每个功能点（F 编号）都有对应的接口契约（§3）
- [ ] 每个接口都有：path/method / 入参表（字段+类型+必填+校验）/ 出参 / 错误码
- [ ] 每个错误码都有：code 常量、含义、触发条件
- [ ] 数据结构（§4）先于接口定义,且接口中所有引用类型都在 §4 出现过
- [ ] 异常与边界（§5）至少覆盖：空状态、首次启动、并发、状态非法、资源缺失
- [ ] 验收用例（§6）覆盖了 §1 所有 F 编号
- [ ] 业务规则（§2）都被某条接口或 AC 引用到（孤儿规则 = 没约束的规则）

**常见漏**：
- 缺错误码（接口只写成功分支）
- AC 写主观描述（"流畅"、"好看"）——必须改成可测子项
- 边界只写"正常运行"情况,异常没列

---

## 二、可落地性

> 检查 spec 在本项目技术栈下"能不能写出来"。

- [ ] 用到的 Tauri command 都在 `capabilities/default.json` 声明了对应 permission
- [ ] 用到的 Tauri event 通道名在前后端都引用同一个常量（不能字符串散落）
- [ ] 单帧尺寸/窗口尺寸约束对得上现有 `tauri.conf.json` 的 `width/height`
- [ ] sprite sheet 路径符合 `public/sprites/<id>_sheet.png` 约定
- [ ] 用到的 Rust 依赖在 `Cargo.toml` 已有（如果新加, 列出来）
- [ ] 用到的 npm 包在 `package.json` 已有（如果新加, 列出来）
- [ ] 涉及文件系统的部分走 `tauri-plugin-fs`（如果有）而不是裸 `std::fs`
- [ ] 涉及 IPC 频繁调用的部分考虑了节流（> 60Hz 的事件要 debounce/batch）

**常见漏**：
- Tauri v2 permission 名称写错（v1 是 `window:allow-start-dragging`, v2 是 `core:window:allow-start-dragging`）
- 假设窗口能"动态改尺寸"但没确认 `resizable: false` 是否要临时改
- sprite sheet 路径大小写不一致

---

## 三、可测性

> 检查 spec 是否"能验证"。每条 AC 必须能 ✓/✗ 客观判断。

- [ ] 所有 AC 都是布尔判断（能做 ✓/✗），不是主观评价
- [ ] 涉及动画/视觉的 AC 拆成"开始时间 / 结束时间 / 不闪黑屏"等可测维度
- [ ] 涉及窗口的 AC 给了具体像素值（"窗口宽 = 240px"）而非"合适大小"
- [ ] 涉及性能的 AC 给了具体数值（"100ms 内"、"< 5MB 内存"）
- [ ] 每条 AC 都标了**怎么测**：手动 / 自动化 / 看 log
- [ ] 没有"应该"、"可能"、"尽量"这种模糊词
- [ ] 错误码对应的 AC 写明了"传什么参数触发"而不是只写"要处理 X 错误"

**常见漏**：
- AC 写"切动画流畅"——改成"切动画 200ms 内完成 + 中间帧不闪白"
- AC 写"不崩溃"——改成"删 X 文件后启动, 30 秒内不死进程"
- 错误码 AC 只写"要处理错误"——改成"调用 `play_animation('不存在')` 应返回 `E_ANIM_NOT_FOUND`"

## 评审意见输出格式

评审时按下面结构输出（用户逐条 ✓/✗ 回复）：

```markdown
### 完整性
- [✓/✗/⚠] <具体意见>
- [✓/✗/⚠] <具体意见>

### 可落地性
- [✓/✗/⚠] <具体意见>
- [✓/✗/⚠] <具体意见>

### 可测性
- [✓/✗/⚠] <具体意见>
- [✓/✗/⚠] <具体意见>

### 总结
- ✗ 项数: N（必须为 0 才能冻结）
- ⚠ 项数: M（可放过但建议跟进）
- 建议: 冻结 / 回 DRAFT 改 X / Y
```

---

## 评审时的"差不多得了"信号

出现下面任一情况, 主动叫停, 要求用户**回 DRAFT 改 spec**：

- 错误码表是空的
- 验收用例 < 功能点数量
- §5 异常场景 < 3 条
- 任一条 AC 含"应该/可能/尽量/不错/还行"
- 业务规则有 1 条没被任何接口或 AC 引用（孤儿规则）
